# Docs

## Criterio

La documentacion del repo debe separarse por objetivo:

- `architecture/`: estructura, convenciones y decisiones tecnicas de alto nivel
- `marketing/`: documentacion propia de la web publica
- `nexo/`: documentacion propia del ERP NEXO AV
- `shared/`: convenciones y patrones reutilizables
- `security/`: hardening, remediacion y checklists
- `sharepoint/`: reglas documentales y operativas de SharePoint
- `important/`: referencias canonicas de negocio que no deben derivar

## Regla practica

Si un documento define como debe organizarse o evolucionar el repo, debe vivir en `docs/architecture/`.
